ACID PROPE:-(ACID properties are specifically for transactions.)(rules that guarantee a transaction is safe and reliable).
1)A-->Atomicity
→ Transaction is all or nothing.

2)C-->Consistency
→ Data remains correct after transaction.

3)I-->Isolation
→ Transactions don’t interfere with each other.

4)D-->Durability
→ Once committed, data is permanently saved.

A - ATOMICITY:-- TRANSACTION (INSERT/DELETE/FETCH/JOIN) -- IT SHOULD HAVE TWO OPTIONS IN TRANSACTION TRUE OR FALSE.
NOTE:- COMPLETELY FAIL (ROLL_BACK) OR DONE COMPLETELY.

C - CONSISTENCY:-- TRANSACTION -- CREATE VIEW / (AFTER TRANSACTION MAIN TABLE SCHEMA (STRUCTURE) SHOULD NEVER BE CHANGED)

	DATATYPE AND NOT NULL, FOREIGN KEY -- ALL SHOULD BE SAME.
	EX:- A HAS 1000RS AND B HAS 500RS THEN IF IN A TRANSACTION A GIVES 200RS TO B THEN AFTER TRANSACTION COMPLETE A SHOULD HAVE 800 AND B 700.IT SHOULD NOT BE LIKE RS DEDUCTED FROM A 	     BUT NOT GOT TO B THEN THIS IS ISSUE OF CONSISTENCY.
	NOTE:-DATABASE GOES FROM ONE CORRECT STATE ->ANOTHER CORRECT STATE.

I - ISOLATION:-- AMAZON (VIVO MOBILE -- ADD_TO_CART -- PAYMENT) -- EACH AND EVERY TRANSACTION MUST BE INDIVIDUAL (PERSONAL).

	MANY PEOPLE CAN BUY SAME PHONE (AMAZON) AT SAME TIME. BUT EVERY TRANSACTION MUST BE PERSONAL AND UNIQUE. MULTI ORDER -- EVERY ORDER INDIVIDUAL.

D - DURABILITY -- IN ANY SCENARIO OUR DATABASE IF HACK/CRASH/HEAT -- THEN TOO TILL LAST ENTRY DATA SHOULD BE BACKUPED. ARCHITECTURE OF DATA MUST BE DURABLE (BACKUPED/LONGLASTING).
